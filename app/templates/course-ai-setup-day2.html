К концу этой инструкции у тебя есть полностью оснащённое рабочее место для вайб-кодинга и полностью настроенный «мозг» твоего проекта, который агент будет читать перед каждым действием. Завтра, на Дне 3, ты по этому мозгу соберёшь работающий сайт.

Время: 90–120 минут. Большую часть делает Claude, ты только говоришь ему, что делать.

Главное правило сегодня: ничего не делай руками. Не создавай папки кнопкой «New Folder», не пиши код в файлах вручную. Всё — через Claude Code. Учишься управлять, а не исполнять.

▶ 🔒 Этап 1. Безопасность Claude Code и звуковые уведомления развернуть

Прежде чем подпускать агента к проекту — подкрутим две важные настройки: звуковой пинг и базовые запреты.

Шаг 1.1. Звуковые уведомления (hooks)

Проблема: ты запустил задачу, ушёл за чаем, Claude закончил через 2 минуты и ждёт твоего ответа — а ты думаешь, что он ещё работает. Полчаса потеряно.

Решение — hooks: звук при завершении задачи и при запросе разрешения. Открой VS Code, открой Claude Code в любой папке (если нет — создай папку moy-proekt через File → Open Folder, дальше всё сделает Claude). В чат:

Промпт для агента
Настрой мне в .claude/settings.json звуковые уведомления (hooks): 1. Звук при завершении задачи (Stop hook) — чтобы я слышал, когда ты закончил работу. 2. Звук при запросе разрешения (Notification hook) — чтобы я слышал, когда ты ждёшь моего «да/нет». Используй системный звук моей операционной системы. Покажи итоговый файл .claude/settings.json и объясни, что ты добавил.

Проверка: задай ему любую задачу («посчитай 2+2 и заверши работу»). Должен пикнуть.

Шаг 1.2. Базовые правила безопасности

Агент — мощный инструмент. Без рамок он может случайно удалить файлы, прочитать секреты, отправить твои данные наружу. Поставим четыре жёстких запрета:

  • rm -rf * — удаление диска
  • export + curl — утечка секретов наружу
  • доступ к .env — API-ключи и пароли в целости
  • wget с переменными — экспорт твоих данных
Промпт для агента
Настрой мне в .claude/settings.json правила безопасности. ЗАПРЕТИ: 1. Команды вида `rm -rf` и любое массовое удаление файлов или папок без явного подтверждения от меня. 2. Связку `export` + `curl` — любую отправку переменных окружения наружу через curl. 3. Чтение файлов .env и любых файлов с секретами, ключами, паролями. 4. `wget` с подставленными переменными — любую отправку моих данных наружу через wget. Используй секцию permissions.deny (или эквивалентную секцию в актуальной версии Claude Code). После изменения — покажи итоговый файл .claude/settings.json и объясни каждое правило одной строкой.

Готово. Теперь даже если ты случайно скажешь «удали всё» — агент остановится и переспросит.

▶ 🧠 Этап 2. Папка .business/ — мозг бизнеса развернуть

Это главный концепт Дня 2. Если ты сделаешь только этот этап и больше ничего — уже будешь сильно впереди.

Зачем. Без бизнес-контекста Claude каждый раз начинает с нуля — не знает твою аудиторию, цены, цели, дизайн. Результат — слабый, шаблонный, «как у всех». С папкой .business/ Claude знает про тебя всё и работает как член твоей команды.

Папка с точкой — это значит скрытая. Она автоматически попадает в .gitignore и не утекает в публичный репозиторий. Внутри лежат твои секретные знания про бизнес.

Шаг 2.1. Архитектура папки .business/

Это пример, не догма. У тебя может быть любая структура — главное, чтобы внутри лежала вся информация про твой бизнес, а INDEX.md объяснял, что где искать.

.business/ ├── INDEX.md ← оглавление, входная точка ├── company/ ← about, team, values, legal ├── products/ ← overview, pricing, product-1, product-2… ├── audience/ ← avatar, segments, objections, journey ├── goals/ ← annual, quarterly, monthly, kpi ├── economics/ ← unit-economics, revenue, costs, forecast ├── marketing/ ← channels, funnel, competitors, content └── assets/ ← brand-guidelines, logo/, testimonials

INDEX.md — карта бизнес-папки. Claude читает её первой, перед любой задачей, связанной с бизнесом. Здесь название проекта, ссылки на все файлы и правило «не загружай всё сразу».

company/

  • about.md — кто мы, миссия в одно предложение, год основания, формат (соло / команда)
  • team.md — кто чем занимается, зоны ответственности, контакты
  • values.md — 3–5 ценностей, которыми руководствуемся при принятии решений
  • legal.md — юр. форма, реквизиты, страна, режим налогообложения

products/

  • overview.md — что мы продаём в принципе, одной картиной
  • pricing.md — все тарифы, что входит в каждый, цены, скидки
  • product-1.md — описание конкретного продукта: задача, фичи, как работает, на каком этапе

audience/

  • avatar.md — портрет идеального клиента: возраст, город, доход, занятие
  • segments.md — какие сегменты ЦА у нас есть и чем они отличаются
  • objections.md — главные возражения и страхи при покупке + как мы их закрываем
  • journey.md — путь клиента: как узнал → как пришёл → как купил → как остался

goals/

  • annual.md — большая цель на год (выручка, аудитория, продукт)
  • quarterly.md — три-четыре цели на квартал
  • monthly.md — фокус месяца, 1–3 пункта максимум
  • kpi.md — какие метрики мы измеряем еженедельно или ежемесячно

economics/

  • unit-economics.md — сколько стоит привлечь клиента, сколько он приносит, маржа
  • revenue.md — структура доходов (откуда деньги)
  • costs.md — структура расходов (куда уходят деньги)
  • forecast.md — прогноз на ближайшие 3–6 месяцев

marketing/

  • channels.md — все каналы привлечения и их состояние (работает / тестируем / умер)
  • funnel.md — воронка: показы → клики → лиды → продажи → удержание
  • competitors.md — 3–5 главных конкурентов, чем мы отличаемся
  • content.md — контент-план и тон коммуникации

assets/

  • brand-guidelines.md — цвета, шрифты, тон
  • logo/ — файлы лого (svg, png)
  • testimonials.md — отзывы клиентов, кейсы, цифры
Шаг 2.2. Создаём .business/ за один прогон

Это самый ценный промпт во всей инструкции. Скопируй его целиком, открой Claude Code в папке твоего проекта, вставь, ответь на вопросы по очереди — и в конце у тебя будет полностью заполненная папка .business/ под твою идею. Никаких «создай папку», «теперь файл» — один промпт делает всё.

Промпт для агента
Ты помогаешь мне создать папку .business/ для моего проекта по методологии вайб-кодинга. Это скрытая папка с бизнес-контекстом, которую ты потом будешь читать перед каждой задачей, связанной с бизнесом. МОЯ ИДЕЯ В ОДНОМ ПРЕДЛОЖЕНИИ: [ВПИШИ СЮДА СВОЮ ИДЕЮ] ДЕЙСТВУЙ ПО ШАГАМ: ШАГ 1. Создай в корне проекта папку .business/ и подпапки: company/, products/, audience/, goals/, economics/, marketing/, assets/. Создай INDEX.md в корне .business/ с заголовком «Бизнес-контекст проекта» и пустым оглавлением (заполнишь в конце). Добавь .business/ в .gitignore (создай .gitignore, если его нет). ШАГ 2. Возьми у меня интервью по семи блокам в таком порядке: company → products → audience → goals → economics → marketing → assets. ПРАВИЛА ИНТЕРВЬЮ: - Иди ПО ОДНОМУ блоку за раз. Закончил блок — записал файлы — перешёл к следующему. - В каждом блоке задавай вопросы ПО ОДНОМУ. - К КАЖДОМУ вопросу сразу предлагай 3-4 варианта ответа на основе моей идеи. Я выберу номер, или скажу «свой вариант: …», или скажу «не знаю». - Если я говорю «не знаю» — предложи свой вариант, который ты считаешь лучшим для моей идеи, и спроси, согласен ли я. - Если я даю короткий ответ — задай 1-2 уточняющих вопроса с вариантами. - Не задавай больше 4-5 вопросов на блок. Не превращай это в допрос. ШАГ 3. После каждого блока СРАЗУ записывай результат в соответствующие файлы внутри подпапки (например, company/about.md, company/team.md и т.д.). Структура файлов внутри подпапок: - company: about, team, values, legal - products: overview, pricing, product-1 - audience: avatar, segments, objections, journey - goals: annual, quarterly, monthly, kpi - economics: unit-economics, revenue, costs, forecast - marketing: channels, funnel, competitors, content - assets: brand-guidelines, testimonials Если у меня в блоке мало данных — создай файл с заголовком и пометкой «заполнить позже», но не пропускай. ШАГ 4. После всех семи блоков обнови INDEX.md так, чтобы он содержал: - название проекта одной строкой, - одно предложение «о чём проект», - список всех файлов с одной строкой описания каждого, - правило: «при работе читай только нужный файл, не загружай всё сразу». ШАГ 5. В самом конце покажи мне дерево созданной .business/ и спроси, что добавить или изменить. ВАЖНО: - Всегда отвечай мне на русском. - Не пиши код, не делай ничего, кроме создания файлов и интервью. - Не задавай все вопросы сразу — только по одному. Начинай с ШАГА 1, потом сразу переходи к ШАГУ 2 и блоку company.
💡 Отвечай агенту вдумчиво. Чем подробнее ответы — тем умнее завтра будет Claude, когда ты будешь собирать сайт.
▶ 📋 Этап 3. Файл CLAUDE.md и папка plans/ развернуть

.business/ — это знания. Теперь поставим правила: как Claude должен с этими знаниями работать.

Шаг 3.1. Создать CLAUDE.md — мозг проекта

CLAUDE.md — главный файл, который Claude читает перед каждым действием в этом проекте. В нём — оглавление, правила работы и ссылки на .business/.

⚠️ Не больше 200 строк — иначе сжигаешь токены каждый запрос.

Скопируйте этот промт целиком и вставьте в окно Клоду:

Промпт для агента
Создай в корне проекта файл CLAUDE.md по этому шаблону: # [НАЗВАНИЕ ПРОЕКТА] — навигация по проекту Карта проекта. Читай в начале каждой сессии: где что лежит и как с этим работать. ## Что это за проект **[НАЗВАНИЕ]** — [одно предложение: что это и для кого]. Подробности — в `.business/INDEX.md`. ## Структура репозитория [НАЗВАНИЕ]/ ├── CLAUDE.md ← этот файл, навигация ├── .business/ ← бизнес-контекст (скрытая папка, не в git) │ ├── INDEX.md ← оглавление, входная точка │ ├── company/ products/ audience/ │ └── goals/ economics/ marketing/ assets/ ├── plans/ ← технические планы (один план = одна функция) └── (код проекта появится позже) ### Где что искать | Что нужно | Куда смотреть | |---|---| | Бизнес-логика, для кого | `.business/INDEX.md` | | Аудитория, боли | `.business/audience/` | | Продукт, тарифы, цены | `.business/products/` | | Цели и метрики | `.business/goals/` | | Экономика | `.business/economics/` | | Каналы, контент, активы | `.business/marketing/` | | Брендинг (лого, цвета) | `.business/assets/` | | Технические планы | `plans/` | ## Ключевые принципы 1. [ПРИНЦИП 1] 2. [ПРИНЦИП 2] 3. [ПРИНЦИП 3] ## Стек [ВСТАВЬ СТЕК — например: Next.js 14, React, Tailwind CSS, TypeScript]. ## Как работать - Перед задачей читай этот файл, затем нужные файлы из `.business/` и `plans/`. - Меняется бизнес-логика → обнови `.business/INDEX.md`. Меняется план → обнови файл в `plans/`. Появилась важная папка → обнови этот CLAUDE.md. ## ВАЖНО: план для каждой новой функции Любая функция оформляется планом в папке `plans/`. 1. Один план = одна функция. 2. Имя файла: `YYYY-MM-DD-название-функции.md`. 3. План делится на фазы со статусом `[ ]` / `[x]`. 4. В конце плана — итог: реализован целиком или нет, что осталось. 5. Любой агент актуализирует план после каждой сессии. ## ВАЖНО: завершение каждого чата В конце сессии пиши рефлексию в `.business/история/YYYY-MM-DD-краткое-название.md` (создавай папку при необходимости): 1. Какая задача. 2. Как решал. 3. Решил — да / нет / частично. 4. Что можно было лучше. 5. Как было и как стало. ## Язык Всегда отвечай мне на русском. Действуй так: 1. Прочитай .business/INDEX.md и все файлы внутри .business/, чтобы понять, что за проект мы делаем. 2. Заполни в шаблоне все поля в квадратных скобках [ВСТАВЬ ...] на основе моих ответов из .business/. 3. ⚠️ В шаблоне уже зашиты два важных правила: «план для каждой новой функции» и «рефлексия в конце каждого чата». После создания этого файла Claude будет следовать им автоматически — потому что читает CLAUDE.md перед каждым действием.
Шаг 3.2. Создать папку plans/

plans/ — папка для технических планов реализации. Один план = одна функция. Завтра, на Дне 3, когда ты скажешь «сделай мне такую-то фичу» — Claude сначала напишет план в plans/, разобьёт его на фазы со статусами [ ] / [x], потом по нему пойдёт и будет отмечать сделанное. Новый чат не помнит, что было сделано. План — помнит.

Промпт для агента
Создай в корне моего проекта папку plans/. Внутри создай файл README.md, в котором коротко (10-15 строк) опиши: 1. Что такое папка plans/ — это место для технических планов реализации функций. 2. Правило: один план = одна функция. 3. Формат имени файла плана: YYYY-MM-DD-название-функции.md 4. Каждый план делится на фазы. У каждой фазы статус [ ] или [x]. 5. В конце плана — итоговый блок: реализован целиком или нет, что осталось. 6. Любой агент, работающий с планом, обязан его актуализировать: отмечать сделанные фазы, дописывать что сделал. После создания покажи дерево корня проекта (CLAUDE.md, .business/, plans/), чтобы я убедился, что всё на месте.
Шаг 3.3. Критика плана — ключевая фишка методологии

Это самое важное место в дне. Любой план, который написал Claude — сырой. Он написан «в один проход», без вопросов, без критики. Правильный цикл:

  • Claude пишет первичный план в plans/YYYY-MM-DD-название.md.
  • Ты копируешь ссылку на этот файл, открываешь новый чат.
  • В новом чате просишь провести глубокое исследование и критику плана через параллельных агентов.
  • Возвращаешься к исходному плану и обновляешь его с учётом критики.
  • Только после этого — реализация.
Промпт для агента
У меня есть первичный план реализации: [путь к файлу, например plans/2026-04-24-landing.md]. Задача: 1. Прочитай план целиком. 2. Прочитай .business/INDEX.md и связанные файлы, чтобы понять контекст проекта. 3. Запусти 3 параллельных субагента на глубокое исследование: - агент 1: проверь техническую реализуемость каждой фазы, найди риски и подводные камни - агент 2: проверь соответствие плана бизнес-целям из .business/, найди пробелы и противоречия - агент 3: сделай ревью UX/пользовательского сценария — где пользователь может застрять, что непонятно 4. Собери их выводы в один отчёт с конкретными правками: что добавить, что убрать, что переформулировать. 5. Покажи мне отчёт. Я решу, какие правки принять, и попрошу тебя обновить файл плана. Всегда отвечай на русском.
💡 Почему это работает: один агент в одном чате не может «посмотреть на свою работу со стороны». Новый чат + параллельные субагенты — это и есть «взгляд со стороны», который ловит 80% проблем до того, как ты начал кодить.
▶ ✅ Этап 4. Проверочный цикл и правила работы развернуть

Самый важный этап. Прямо сейчас проверим, что вся машина работает.

Шаг 4.1. Тест-цикл

Дай Claude микро-задачу и проверь, что он читает CLAUDE.md и .business/INDEX.md перед началом, создаёт план в plans/, а в конце пишет рефлексию.

Промпт для агента
Тест-задача: создай в корне проекта файл HELLO.md с одной строкой «Привет от [подставь название проекта из .business/INDEX.md]». Действуй полностью по правилам из CLAUDE.md: 1. Сначала прочитай CLAUDE.md и .business/INDEX.md. 2. Создай план в plans/ с фазами и статусами (даже для такой простой задачи). 3. Выполни задачу. 4. Обнови план — отметь фазы как [x]. 5. В конце сессии запиши рефлексию в .business/история/ по формату из CLAUDE.md. Если что-то непонятно — спроси.

После выполнения проверь руками: появился ли HELLO.md в корне? появился ли план в plans/? появился ли файл рефлексии в .business/история/? Если что-то не появилось — значит, CLAUDE.md написан плохо. Попроси Claude доработать его и повтори тест.

Шаг 4.2. Семь правил работы с агентом

Выучи эти семь правил как мантру. Это разница между тем, кто бьётся 5 часов с одной фичей, и тем, кто делает её за 30 минут.

  • Один чат — одна задача. Закончил → закрой чат → открой новый. Очищает контекст и твою «оперативку».
  • Контекст ≤ 60%. Если окно заполнено больше — выполни /compact или /clear. Иначе агент тупит.
  • Ошибки чини в том же чате, в котором писал функцию. Не открывай новый — потеряешь контекст.
  • Долго думает (5–10 мин) — НЕ перезапускай сразу. Подожди. Если совсем висит — /clear и сформулируй задачу короче.
  • Ничего не делай руками. Папки, файлы, команды — всё через Claude.
  • Каждая функция = план в plans/ (уже зашито в CLAUDE.md).
  • Каждый чат заканчивается рефлексией в .business/история/ (тоже зашито в CLAUDE.md).

Промпт для агента

Найди 10 причин, из-за которых мы обосрёмся: - где в CLAUDE.md двусмысленности, которые ты как агент можешь понять неправильно - где в .business/ пробелы, из-за которых решения придётся выдумывать - где бизнес-логика противоречит сама себе - где стек или принципы противоречат реальности моей идеи - где я переоцениваю объём одного дня (типа «соберём маркетплейс за день») Не стесняйся. ▶ 🛠 Этап 5. Скиллы развернуть

Скилл — это папка с инструкцией, которую Claude подтягивает сам, когда видит подходящий триггер. Один скилл умеет делать дизайн, другой — парсить PDF, третий — проводить 12-этапный воркфлоу от ресёрча до деплоя.

Скилл — это папка в .claude/skills/название-скилла/. Внутри: SKILL.md (сохранённый промпт-инструкция + триггер) и опциональные файлы (шаблоны, примеры, скрипты). Когда ты пишешь задачу под триггер — Claude сам подгружает содержимое скилла.

Два уровня установки: глобальные (~/.claude/skills/) — во всех проектах; локальные (.claude/skills/ в корне проекта) — только в этом проекте.

🎯 Правило: ставь локально, если скилл нужен конкретному проекту. Глобально — только то, чем реально пользуешься везде. Чем меньше скиллов — тем меньше токенов на их триггеры.
⚠️ Безопасность скиллов. Скилл — исполняемая инструкция от автора. Плохой может прочитать твой .env, отправить ключи на внешний URL, удалить файлы. Перед установкой ЛЮБОГО скилла: открой репозиторий → прочитай README и SKILL.md → поищи опасные паттерны (curl, wget, .env, rm -rf) → проверь звёзды и автора. Ставь от Anthropic или из крупных проверенных каталогов (Smithery).
Промпт-аудит любого скилла
Я хочу установить скилл по ссылке [ВСТАВЬ URL РЕПОЗИТОРИЯ]. Скачай его исходники, прочитай содержимое (README.md, SKILL.md, все .md и .sh файлы) и проверь, нет ли там опасных операций: - чтения .env или других файлов с секретами - отправки данных на внешние URL (curl, wget, fetch) - удаления файлов (rm, unlink) - запуска внешних бинарников - подмены существующих файлов проекта Дай вердикт: безопасно / подозрительно / опасно — и объясни почему по пунктам. Если опасно — НЕ ставь.

▶ 🌐 Этап 6. Playwright — Claude управляет браузером развернуть

Playwright MCP — одна из главных «вау»-фишек вайб-кодинга. Claude открывает у тебя настоящий браузер и сам тыкает на кнопки, заполняет формы, делает скриншоты, проверяет мобильную вёрстку: проверяет твой лендинг, тестирует чужой сайт, гоняет регрессию после правок.

⚠️ У некоторых Playwright работает нестабильно, у других — идеально. Если полезут странности — не зависай, поставь альтернативу (Chrome DevTools MCP) и иди дальше.
